NodeBB

  • Fortunately I found another cool forum software called nodebb. It has everything that discourse has plus it's written in node so I don't have to learn a new language in case I want to hack it. Furthermore I really liked it's plugin api so I decided to go with it.
  • I can use the nodebb write api to create a new user
  • It looks like we can activate nodebb plugins via command line
  • The issue that added command line setup script for nodebb
  • The install directory contains default setup data that can be configured to set different defaults for the type of forum user we are targeting. ex a community of landlords may have different defaults from say a community of college students.
  • The default setup with no parameters automatically creates an adminstrative account with a username and a randomly generated password.
  • Nodebb requires restart everytime a new plugin is installed or a theme is changed.
  • By default nodebb installs with a pre-configured list of topics and categories. These topics and categories are defined statically in the /install/data directory. At the time of installation we should be able to modify the files contained in this directory to create categories and topics specific to our particular audience. For example the categories and topics for a college community forum would be a different as compared to the categories and topics for a boutique hotel helpdesk app.
  • Node bb can setup via environment variables. Once the app is setup it creates a config.json file. If we include a config.json file with the docker image we won't even need to bother with the setup steps.

Application architecture of the forum_container

The forum_container is a git branch which represents an instance of the docker container spawned from the base image, consisting of the following applications:-

  1. A redis server
  2. Nodebb forum running on port number 4567 along with all it's plugins and node_modules
  3. Openresty as the reverse proxy server

Once the container has been initialized all the updates to it will be pulled dynamically from the forum_container branch. Therefore we need to configure our applications to:-

[we might not need all these steps as docker swarm could manage rolling updates for us]

  1. Communicate with the master process. Master process should be able to send instructions to the running application. And the running application must be able to execute it.
  2. The process of communication must be two ways. The application must also be able to send updates to the master process about it's state.
  3. Since all the state information will be stored in the git repository we need to configure our app to pull from the branch at runtime, apply the changes and start the server again. Fortunately there are node bindings for native git client that should simplify this process a lot.

  2. A knowledge base is a topic with no replies. Can be pinned (or stickied) possibly.
  3. Community forums are just vanilla forums
  4. And helpdesk is #1 and #2 combined.
