Top ten things about openresty

  1. It's nginx we know and love

    Say you are the developer of one of those 23 millions websites [1] that use nginx as their webserver and you see this shiny new thing called openresty and you say to yourself

    "Bah! HUMBUG! I can't afford to break my pretty nginx configruations!! Humbug! Bah!"

    Ah yes. I understand the feeling pretty well. It's a dilemma faced by many programmers who hear about operesty for the first time. "Should we migrate or should we stay with nginx?" But they miss the point. Openresty is not a fork of nginx. It is nginx. Or maybe I should say it is nginx +

    Bundled together with some well tested third party modules and integrated with luajit it gives the developers the ability to program nginx. Yes I said program. You are not just limited to configuring the webserver (although you can still use it that way) but you can actually write code. And we are not talking about some wierd pseudo language that nginx gave you by default. We are talking about a real programming language -- lua. And that's about it. Nothing else changes.

    So let me type this in strong -- openresty is nginx-- and this in small -- with extra goodies packaged together in a nice little bundle. Your existing configuration will continue to work with openresty. You might need to do some additional work if you've installed third party modules. Even then there is a pretty good possibility that your favorite third party nginx module comes budled with openresty. In the worst case scenario you might need to install those bundles again but the default stuff works out of the box. And that makes for a damn smooth migration.

  2. Fast

    3. I know nginx is fast. You know nginx is fast. Everybody knows nginx is fast. The question is "Is it still fast with lua?"

    Yes it is, it seems . openresty uses luajit to compile lua code. The code compiled with lua jit is damn fast. And you've got one lua jit vm per nginx worker process.

  5. *by_lua directives

    6. "Lua code inside strings is ugly yuck! yuck! yuck!"

    Calm down old fellow. I belive that saying yuck! one time is quite enough to convey the feeling of repugnance that a programmer feels when he has to write code embbded in strings.

    I will concede that emotions run high when the debugger fails to tell at exactly what line the error has occured. And in this situation it is quite okay to say yuck! yuck! to emphasize that the disgust is quite strong.

    But in my opnion saying yuck! yuck! yuck! three times in quick succession helps no one. I mean to say that the feeling of strong disgust has already been conveyed by saying "yuck!" two times and a third utterance of yuck! only weakens the effect. Infact I believe that it would be far more powerful to say "yuck" and just leave it hanging there. Don't even use an exclamation mark to close it off. Imagine Alan Rickman saying Yuck and try to capture that effect when you speak.

    With that out of the way... you don't need to use strings to write lua code. You can have your all your lua in a file and refrence to that file using the *by_lua_file. All *by_lua directives have an equivalent *by_lua_file directive as well so you are not really bound to single choice. That clears up the clouds I hope :)
