Top ten things about openresty

  1. It's nginx we know and love

    Say you are the developer of one of those 23 millions websites [1] that use nginx as their webserver and you see this shiny new thing called openresty and you say to yourself

    "Bah! HUMBUG! I can't afford to break my pretty nginx configruations!! Humbug! Bah!"

    Ah yes. I understand the feeling pretty well. It's a dilemma faced by many programmers who hear about openresty for the first time. "Should we migrate or should we stay with nginx?" But they miss the point. Openresty is not a fork of nginx. It is nginx. Or maybe I should say it is nginx +

    Bundled together with some well tested third party modules and integrated with luajit it gives the developers the ability to program nginx. Yes I said program. You are not just limited to configuring the webserver (although you can still use it that way) but you can actually write code. And we are not talking about some wierd pseudo language that nginx gave you by default. We are talking about a real programming language -- lua. And that's about it. Nothing else changes.

    So let me type this in bold -- openresty is nginx-- and this in small -- with extra goodies packaged together in a nice little bundle. Your existing configuration will continue to work with openresty. You might need to do some additional work if you've installed third party modules. Even then there is a pretty good possibility that your favorite nginx module comes budled with openresty. In the worst case scenario you might need to install those bundles again but the default stuff works out of the box. And that makes for a damn smooth migration.

  2. Fast

    3. I know nginx is fast. You know nginx is fast. Everybody knows nginx is fast. The question is "Is it still fast with lua?"

    Yes it is, it seems . openresty uses luajit to compile lua code. The code compiled with lua jit is damn fast reaching alomst c levels of performance.

  3. Low memory footprint

  4. Lua

    5. I know that nignx... you know that nginx... everybody knows that nginx... wait this won't work here. Let's try something else.

    Openresty uses lua as the programming language. Now lua has some pretty cool things like the ability to return multiple values from a function (which is used brilliantly by the way in the ngx.location.capture api), declaring multiple variables on a single line, co routines to pause and start the execution of functions etc. It has a very familliar syntax and decent amount of third party modules.

    But there are a couple of things that can be off putting to a person just starting out. Like for example arrays that start with 1. No ternanry operators. For some people it can be a bit too verbose with all those end statements. But these "problems", if you want to call it that, is just a matter of getting used to the language. The only serious difficulty that I had while I was learning lua was getting my head around 1 based array indices. Once you get used to lua,which should be pretty quick anyways, you wouldn't mind these petty things.

    Lua makes sense as a language of choice for openresty because it's highly embeddable and it has luajit. Which as we all know is damn fast and damn conversvative when it comes to resrouce usage. Hot Damn!

  5. *by_lua directives

    6. You get to use all the sweet goodness of lua using the *by_lua directives of openresty. These directives do diffrent things depending upon the context they are run in. The lua_code_cache directive for instance turns the caching of lua code on/off. That might increase the perfomace of our program but it does not let us sample the sweet goodness of lua in nginx.

    For that consider access_by_lua directive which allows you to do some lua processing during the access phase of an nginx request. Or rewrite_by_lua directive that allows you to inject lua into the nginx rewrite rquest phase.

    Cool right? At the very least these directives should give you a good excuse to learn more about the http request processing phases of nginx so you can inject lua code and hack it.

All *by_lua directives have an equivalent *by_lua_file directive as well so you are not really bound to single choice.   6. ngx.location.capture.multi --the ultimate api mashup tool

    7. ngx.location.capture is easily my favorite thing about openresty. In simple terms it allows you to make synchronous yet non blocking subrequests to nginx's location blocks.

    "Okay so what's so cool about it?"

    Well here's what's cool. You know how location blocks can use proxy_pass directives to pass requests to different servers right? With location capture api you can get a response from the proxied server in your code. The implication of this is that you can have multiple services deployed in different locations and make requests from them

    Think microservices. Let's say your application has different services for authentication, billing and notification operations.You can have these services proxied with nginx locations and access them cleanly in lua with location.capture.

    The good thing about proxying through locations is that you can use the same method to proxy both external and internal systems. For example say you are bulding a system that requires calling external apis of mailchimp, paypal and amazon. To make these api's available all you need to do is configure thier respective location blocks. And that's it. You can now start making requests to them from your lua code.

  7. Simplified application architechture

    8. The lua directives offer a clean speration of the request coming in to nginx. Well to be precise nginx has a well defined cycle which a request must go through before a final response is sent. What openresty offers is the ability to inject lua code into into these request phases.

    So for example an access_by_lua handler can allow you to inject lua while nginx process a request in an access phase. Effectively what this does is it allows you to write aribitray code that can reject a request then and there before it moves on to other handlers. This is very useful if you want to reject unauthorized access to a location block or configure a blacklist of ip addresses or impose other restrictions on the service of a request. 

                
location / {
access_by_lua_block {	      
local headers = ngx.req.get_headers()
local uname_pass = 'abc:def'
if  headers["Authorization"] ~= "Basic "..ngx.encode_base64(uname_pass) then
   ngx.header["WWW-Authenticate"]="Basic"
   ngx.req.clear_header("Authorization")
   return  ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
ngx.exit(ngx.OK)
 }
}

    "But exactly how does it simplify the architechture"

    First the practice of code reusablity is highly encouraged. Once you have written an access rule you can reuse the same code by refrencing it from different access_by_lua file.

    Second it allows you to think of your code in terms of request processing phase handlers cleanly breaking down each action and tying it down to a particular handler. So your authorization/authentication code will be handled by the access_directive.

  8. A good mix of configurability and programability

    9. A definitive advantage that you get with nginx is the number things that you can do with the modules that are already available. Being a proxy server with one or two lines of configuration you can set block lists for ip addresses,route requests to different handlers. Configure the security headers for your applicaiton. Openresty allows you to do more or perform things that were not possible or difficult to do with just configuration directives.

  9. Unified proxy and Application servers

    10. Where as before you had to maintain different proxy and applicaton servers openresty gives you an opportunity to combine your application and proxy requirements into a single server.

    On the other hand if you already have an existing application server openresty still gives you the benifit of offloading some of the minor but resource consuming tasks to it and take some pressure off of your main servers. So by using openresty instead of nginx you get to have more options.

  10. Concurrency

    11. Saved the best for the last. Openresty is higly concurrent. Each nginx worker process has comes with an instance of lua jit to compile the lua code. The content handlers themselves are run in light threads. So there can be multiple execution points per lua jit instance making it very difficult to write blocking code in openresty.
